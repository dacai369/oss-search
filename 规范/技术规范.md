# 技术规范（SPEC）

## 1. 流水线总览

```
需求输入 → 意图解析·能力拆解(LLM) → 查询扩展 → 多源并行采集(可插拔适配器)
        → 归并去重·实体对齐 → 质量评级(多维) → 排序推荐 + 选型报告
```

## 2. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 语言 | **Python 3.11+** | 爬取 / LLM / 数据生态最全 |
| LLM | **宿主 agent 提供**（不绑定 provider） | skill 跑在谁的环境用谁的模型（见 ADR-005）|
| Embedding | **MVP 用 BM25 降级**（见 ADR-006） | 不引入 embedding 依赖 |
| 语义匹配 | embedding + **pgvector** | 不上重型向量库，够用 |
| 存储 | **SQLite（MVP）→ Postgres** | MVP 单机起步，后期再迁 |
| 任务调度 | 先同步串行，V2 再上队列 | 避免过度工程 |
| 入口 | **CLI（MVP）→ Web** | 先跑通链路 |

> 原则（遵循全局指令）：快速原型、避免过度抽象、不加无用依赖。

## 3. 决策记录（ADR）

### ADR-001 意图拆解：提示词优先，不上 Agent
- **决策**：用结构化 Prompt + 固定 JSON Schema 实现意图拆解，**不用** 自主 Agent。
- **理由**：拆解是有界的一次性转换，无需多步工具调用；Prompt 方案可测试、便宜、稳定。
- **升级路径**：模块对外暴露 `parse_intent(request) -> IntentSpec` 接口，未来内部可换成 Agent（先搜→看结果→改查询的循环），调用方不变。

### ADR-002 内容源：定向白名单，不爬全微信
- **决策**：内容平台（公众号等）只采集 [`信源清单.yaml`](信源清单.yaml) 里的白名单源。
- **理由**：微信无开放搜索 API，全量爬取反爬严重、质量参差；白名单质量高、稳定、可增量。
- **优先吃现成发现型信源**：HelloGitHub、阮一峰科技爱好者周刊、GitHubDaily 等本身就在做项目发现。
- **MVP 不做公众号**，先用有 RSS/友好接口的源（HelloGitHub、掘金等）验证链路。

### ADR-003 技术栈：Python + SQLite 起步
- 单机能跑通，再谈分布式与队列。

### ADR-004 MVP 范围
- 只接 GitHub + npm/PyPI，入口为 CLI，评级用基础加权模型。

### ADR-005 LLM 由宿主 agent 提供，引擎不绑定 provider（已修订）
- **背景修订（2026-06-30）**：本项目以 Skill 形式分发，运行在团队各自的 agent/IDE 里（Claude Code / Cursor / …）。**不应指定也不应要求某个 LLM 或 key**——skill 跑在谁的环境就用谁的模型。
- **决策**：把"动脑"与"干活"分离：
  - **需 LLM 的步骤**（意图拆解、评级理由、组合建议）→ **由宿主 agent 完成**，指令写在 SKILL.md，agent 按《意图拆解Schema》产出 JSON。
  - **确定性步骤**（采集 / 归并 / 数值算分 / 渲染 HTML）→ **引擎脚本**，无需 LLM，可测可复现。
- **agent↔引擎契约** = `IntentSpec` JSON（schema + 校验在 `models.py`）。引擎主路径 = `load_intent_spec(json)` 校验加载。
- **可选兜底**：脱离 agent 单跑/CI 时，引擎可用一个**通用 OpenAI 兼容客户端**（`LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL` 配置，DeepSeek 只是默认示例）。此为可选，团队成员用 skill 时**不需要**配任何 key。

### ADR-006 Embedding 方案待定（阻塞项）
- **问题**：评级里 `relevance`（权重 0.30，最高）依赖 embedding 算需求↔README 相似度，但 DeepSeek 没有成熟 embedding API。
- **候选**：
  1. **本地模型**（如 `bge-small-zh` / `bge-m3`，sentence-transformers）—— 免费、中文好、可离线，需装模型。
  2. **第三方 embedding API**（如硅基流动 SiliconFlow、智谱、OpenAI）—— 省事，需另一个 key。
  3. **MVP 先降级**：relevance 暂用关键词/BM25 重合度近似，V2 再上 embedding。
- **状态**：✅ 已定（2026-06-30）——**MVP 采用方案 3（BM25 降级）**，V2 再上本地 bge。Embedding 依赖暂不引入。

### ADR-007 接口层 = Skill 封装 + 静态 HTML 报告
- **决策**：引擎解耦为独立 Python 库/CLI（TASK-001~008）；MVP 接口用 **Claude Skill 封装**；交付物除 Markdown 外，额外产出**自包含 HTML 报告**（可分享/存档）。
- **不做**：纯前端静态页当引擎（密钥泄露/CORS/限流，技术不可行）；全栈 Web 应用推迟到 V3。
- **分层**：
  ```
  核心引擎(Python 库/CLI)
    ├─ Skill 封装  → MVP 入口（TASK-009）
    ├─ 静态 HTML 报告 → 产出物（并入 TASK-007）
    └─ Web 应用 → V3，按需
  ```
- **约束**：引擎不得依赖 Skill 运行环境，必须能脱离 Claude 单独 `python -m` 跑（保证可移植 + 可测）。

### ADR-008 信源解析 = 内置白名单 + 运行时自定义源
- **决策**：采集信源 = `信源清单.yaml`（内置白名单）∪ **运行时用户自定义源**（URL / RSS / 公众号链接）。
- **实现**：新增一个**通用源适配器**（GenericUrlAdapter），能吃任意 URL/RSS，从中抽取被提及的开源项目并回链仓库（复用 TASK-004 的提及→项目映射）。
- **入口**：自定义源作为 skill/CLI 的一等参数传入（如 `--source URL ...` 或对话里给）。
- **HTML 报告的边界**：静态 HTML 是产出物，**不能**在页面内发起爬取（密钥/CORS/限流）。所以：
  - MVP：报告里**展示**本次用到的自定义源；
  - 交互式"输入网址即重跑"的表单 → 需后端，归入 V3 Web 应用。

### ADR-009 打包与分发
- **决策**：作为 GitHub skill 分发，但因捆带 Python 引擎，**非零配置**（需 `pip install`）。分发包含：SKILL.md + 引擎 + 安装脚本 + 首次配置向导 + README。
- **LLM 无需 key**（由宿主 agent 提供，见 ADR-005 修订）。仅以下为可选：
  - `GITHUB_TOKEN`：建议配，仅用于提高 GitHub 采集限流，**与 LLM 无关**；
  - `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL`：仅当脱离 agent 单跑兜底时才需要，团队用 skill 时不配。
- **安装路径**：`install-skill`（GitHub→多工具，自动转格式）/ 手动放入 skills 目录；装后 `pip install` 引擎依赖即可。

## 4. 核心数据结构

### 4.1 IntentSpec（意图拆解输出）
见 [`意图拆解Schema.md`](意图拆解Schema.md)。

### 4.2 Candidate（候选项目，采集层统一输出）
所有适配器必须输出此结构：

```python
{
  "id": "github:owner/repo",      # 全局唯一，source:identifier
  "source": "github",             # github|gitee|npm|pypi|content
  "name": "repo",
  "full_name": "owner/repo",
  "url": "https://...",
  "description": "...",
  "language": "Python",
  "license": "MIT",
  "stars": 12000,
  "forks": 800,
  "last_commit_at": "2026-05-01T...",
  "open_issues": 30,
  "contributors": 45,            # 可选，拿不到填 null
  "topics": ["search", "llm"],
  "readme_excerpt": "...",        # 用于语义契合度
  "matched_capability": "cap-1",  # 命中的能力 id
  "raw": {}                       # 原始返回，调试用
}
```

> 拿不到的字段填 `null`，不要伪造。评级层按缺失降权。

### 4.3 ScoredCandidate（评级层输出）
在 Candidate 基础上加：

```python
{
  "scores": {
    "activity": 0.0-1.0,
    "community": 0.0-1.0,
    "maturity": 0.0-1.0,
    "security": 0.0-1.0,
    "license_fit": 0.0-1.0,
    "relevance": 0.0-1.0      # 需求 embedding 与 readme 相似度
  },
  "total_score": 0.0-1.0,     # 加权汇总
  "grade": "A|B|C|D",
  "reasons": ["活跃度高：近90天120次提交", "..."]  # 每个分给理由
}
```

## 5. 模块接口约定（契约，各任务卡必须遵守）

```python
# 意图拆解
def parse_intent(request: str) -> IntentSpec: ...

# 采集适配器（统一接口，可插拔）
class SourceAdapter(Protocol):
    name: str
    def search(self, capability: Capability, spec: IntentSpec) -> list[Candidate]: ...

# 归并去重
def dedup(candidates: list[Candidate]) -> list[Candidate]: ...

# 评级
def score(candidates: list[Candidate], spec: IntentSpec) -> list[ScoredCandidate]: ...

# 报告
def build_report(scored: list[ScoredCandidate], spec: IntentSpec) -> str: ...
```

> **接口先于实现**：任何任务卡改动这些签名，必须先更新本规范并经指挥官确认。

## 6. 质量评级维度与权重（MVP 初版，可调）

| 维度 | 权重 | 信号 |
|---|---|---|
| relevance 契合度 | 0.30 | 需求 vs README/描述 语义相似度 |
| activity 活跃度 | 0.20 | 近90天提交、issue 响应 |
| community 社区 | 0.15 | stars / forks / 贡献者 |
| maturity 成熟度 | 0.15 | release 数、版本号、文档/测试存在 |
| license_fit 许可证 | 0.10 | 是否符合约束 |
| security 安全 | 0.10 | 已知 CVE、依赖健康（V2 接 OSV） |

评级：A ≥0.8，B ≥0.65，C ≥0.5，D <0.5。
